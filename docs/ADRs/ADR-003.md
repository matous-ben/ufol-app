# ADR-003: Spring Security vs vlastní řešení / OAuth

## Status
Accepted

## Kontext
Pro řešení autentizace a autorizace admina bylo na výběr 
z následujících přístupů:

- **Spring Security** — framework integrovaný v Spring Boot 
  ekosystému, využívá session-based autentizaci
- **Vlastní řešení** — implementace přihlášení, session 
  managementu a ochrany URL od nuly
- **OAuth2** — přihlášení přes třetí stranu (Google, GitHub, 
  Microsoft). Odpadá nutnost ukládat hesla, ale vzniká 
  závislost na externím poskytovateli a výrazně složitější 
  integrace

## Rozhodnutí
Zvolen framework Spring Security.

## Důvody
Spring Security automaticky řeší vše co by vlastní 
implementace vyžadovala napsat ručně:

- **BCrypt hashování hesel** — plain text heslo nikdy 
  nevstoupí do databáze, framework zajišťuje bezpečné 
  ukládání i porovnávání při přihlášení
- **Session management** — generování náhodného session ID, 
  ukládání session na serveru a nastavení cookie 
  v HTTP response probíhá automaticky
- **Ochrana URL** — místo kontroly přihlášení v každém 
  admin Controlleru zvlášť stačí jedna konfigurace 
  v SecurityConfig.java pokrývající celý /admin/** prostor
- **CSRF ochrana** — automatická ochrana před 
  Cross-Site Request Forgery útoky včetně integrace 
  s Thymeleaf formuláři
- **Automatické odhlášení** — session timeout 
  konfigurovatelný jedním řádkem

Vlastní implementace všech výše uvedených bodů by 
představovala 2–3 týdny práce navíc s vysokým rizikem 
bezpečnostních chyb. OAuth2 je pro jediného interního 
admina zbytečný overkill — přidává závislost na třetí 
straně bez odpovídajícího přínosu.

## Důsledky

**Pozitivní:**
- BCrypt hashování, session management, cookie handling 
  a CSRF ochrana jsou zajištěny frameworkem bez 
  vlastního kódu
- Automatické odhlášení po neaktivitě konfigurovatelné 
  jedním řádkem
- Znalost Spring Security přímo uplatnitelná 
  v enterprise Java projektech

**Negativní:**
- Strmější křivka učení — konfigurace vyžaduje pochopení 
  konceptů SecurityFilterChain, UserDetailsService 
  a AuthenticationProvider
- Výchozí chování frameworku (ochrana všech endpointů) 
  vyžaduje explicitní konfiguraci pro povolení 
  veřejných stránek

**Do budoucna:**
- Přidání registrace fanoušků nebo OAuth přihlášení 
  přes Google je možné bez změny základní architektury — 
  Spring Security tyto scénáře nativně podporuje
